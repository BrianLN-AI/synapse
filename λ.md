## I. Primitive Definitions

*These are the undefined terms — the bottom. Everything in this document is built from them.
They are defined using only concepts that require no prior context to understand.*

### I.1 Logic

```scheme
;; The universe has distinguishable states.
true   ;; the state that obtains
false  ;; the state that does not obtain

(not true)  → false
(not false) → true
(and a b)   → true iff both a and b are true
(or  a b)   → true iff at least one of a, b is true
```

### I.2 Equality

```scheme
;; Two expressions are equal when they denote the same value.
(= a b) → true iff a and b denote the same value

;; Three kinds of equality used in this document:
;;
;;   Referential equality:   (= a b) by address — same blob in the vault
;;   Structural equality:    (= a b) by content — same bytes
;;   Behavioral equality:    (= a b) by outputs — same result for all inputs
;;
;; For blobs: referential = structural (invariant I, Section II).
;; Behavioral equality is a separate predicate — not encoded in =.
;; When this document uses =, it means referential/structural unless stated otherwise.
```

### I.3 Data

```scheme
;; Values
atom    ;; an irreducible unit — a string, number, boolean, or symbol
(list a b c ...)  ;; an ordered collection of values
(cons head tail)  ;; a pair — the primitive from which lists are built

;; A blob is a triple: (type, payload, address)
;;   type    — a string naming what the payload represents (e.g., "logic/python")
;;   payload — the content: arbitrary bytes
;;   address — the canonical identifier (defined in I.5)
```

### I.4 Functions

```scheme
;; A function transforms an input into an output.
(lambda (x) body)      ;; a function that binds x and evaluates body
(apply f x)            ;; invoke function f with argument x
((lambda (x) body) v)  ;; β-reduction: substitute v for x in body

;; All computation in this system is function application.
;; A blob's payload is a function. Invoking a blob is applying that function.
;; The system's entry point is a function. The session is its argument.
```

### I.5 Hash

```scheme
;; A hash function maps content of any size to a fixed-size string.
;;
;;   (hash payload) → fixed-length-string
;;
;; Required properties:
;;   Determinism:        (= payload-a payload-b) → (= (hash payload-a) (hash payload-b))
;;   Collision resistance: (= (hash a) (hash b)) → (= a b) [practically]
;;   One-way:            knowing (hash payload) reveals nothing about payload
;;
;; The canonical vault hash function is blake3.
;; The canonical ZK-circuit hash function is Poseidon.
;; Both are named in the address (I.7) — the function is not assumed globally.
```

### I.6 Content and Address

```scheme
;; content : blob → payload
;;   Returns the payload a blob was created from.
;;   content is recoverable from the vault given an address.

;; address : blob → string
;;   Returns the canonical identifier of a blob.
;;   Two blobs with the same address are the same blob.
;;   An address is computed from the content — never assigned externally.
```

### I.7 Multihash — The Self-Describing Address

```scheme
;; An address is NOT just a hash digest.
;; An address is a self-describing tuple:
;;
;;   address = (encode algorithm digest-length digest)
;;
;;   algorithm      — which hash function produced this digest (e.g., "blake3", "poseidon")
;;   digest-length  — the length of the digest in bytes
;;   digest         — the hash of the content
;;
;; Written:  blake3:af1349b9f5f9a1a6a0404dea36dcc9499bcb25c9adc112b7f9f403afd2d309222
;;           ^^^^^^ ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
;;           prefix digest (64 hex chars = 32 bytes for blake3)
;;
;; Why: an address that omits the algorithm assumes all addresses use the same function.
;; That assumption is false the moment a second hash function enters the system.
;; Leave the address in roots. (See: "Leave the invariants in roots", Section II.)

(define (multihash algorithm payload)
  (string-append algorithm ":" (hash-with algorithm payload)))
```

### I.8 The Vault

```scheme
;; The vault is an append-only collection of blobs.
;;
;;   vault-put  : vault × type × payload → address
;;   vault-get  : vault × address → blob (or not-found)
;;   vault-list : vault × type × predicate → (list blob)
;;
;; The vault has no knowledge of governance, capabilities, or objectives.
;; It stores and retrieves. The vault does not interpret.
;;
;; Append-only means:
;;   A blob that enters the vault cannot be altered or removed.
;;   A blob can be superseded (a new version promoted), but the old version persists.
;;   The vault is the audit trail. The audit trail is the vault.
```

### I.9 Append-Only and the Direction of Time

```scheme
;; (append-only vault) is not a property we choose. It is a consequence of hashing.
;;
;; If a blob could be altered after being stored:
;;   its content would change → its hash would change → its address would change
;;   → the original address would point to nothing, or to a different blob
;;   → any artifact that referenced the original address is now broken
;;
;; Mutability and content-addressing are structurally incompatible.
;; The vault is append-only because content-addressed storage cannot be otherwise.
;; This is not a rule. It is a derivation.
```

## III. Content Addressing

*The vault is the shared substrate. Any agent that can read and write blobs
can participate in the system. The vault does not know what it stores.*

```scheme
;; [ALGEBRAIST]
;;
;; The vault is a category.
;;   Objects:   blobs (typed, content-addressed)
;;   Morphisms: promotions (governed state transitions — old blob superseded by new)
;;   Identity:  a blob promoted to itself (no-op; records a governance decision)
;;   Composition: predecessor chains compose; the audit log records all promotions

(define (put vault type payload)
  (let* ((addr (multihash "blake3" payload))
         (blob (record type payload addr)))
    (vault-append vault blob)
    addr))          ;; returns the address — the only handle the caller gets

(define (get vault addr)
  (vault-lookup vault addr))   ;; returns the blob or not-found
```

```
[BUILDER]

The vault is the foundation. Foundations do not know what is built on them.
A foundation that interprets the building has confused its role.

Load-bearing property: content addressing means the foundation cannot be
undermined silently. A changed blob has a different address. There is no
quiet corruption. Change is always visible.

The ABI contract, governance expressions, GOKR blobs —
these are the building. The vault is the ground.
```

## VI. Self-Modification

*The system improves itself by applying itself to itself.
The governance mechanism that applies to session-init is the same as for any blob.*

```scheme
;; [MATHEMATICIAN] — The fixed-point structure
;;
;; φ = 1 + 1/φ                    (the golden ratio — stable through self-reference)
;; f_n = f_{n-1}(f_{n-1})         (the fabric — same structure)
;;
;; Both are left in roots.
;; Approximating φ as 1.618 loses the self-similar property.
;; Approximating f_n as "the system improves itself" loses the fixed-point structure.

(define (evolve label goal contract governance)
  (let* ((candidates  (generate-n label goal 3))      ;; 3 candidates per cycle
         (adversarial (test-adversarially candidates contract)) ;; separate tester model
         (winner      (argmax fitness adversarial))
         (approved    (govern winner governance)))
    (when approved
      (vault-promote label winner)
      (audit-append  label winner goal))))

;; Key: evolve is itself a blob.
;; The upgrade path for evolve: put a new evolve blob, propose it, test it,
;; benchmark it, promote it. No special case. No out-of-band upgrade.
;; This is invariant IV made operational.
```

```
[BUILDER] — Tensegrity upgrade

A tensegrity structure holds its shape through balanced tensions, not rigid supports.
When one element is replaced, the tensions redistribute.
The structure does not stop functioning during the upgrade.

Upgrade protocol:
  1. Identify the element to replace (the label)
  2. Generate candidate replacements
  3. Verify each satisfies the tension relationships (ABI contract)
  4. Select the replacement that best maintains or improves balance (fitness)
  5. Govern the replacement (the structure approves its own modification)
  6. Promote — the old element persists in the vault; the new one is canonical

The governance mechanism is the tensegrity. Remove it and the structure collapses.
```

  These live in the current goal/gokr blob. They evolve. This document does not.

- **Installation and runtime requirements** — dependencies, Python version, how to run.
  These are mortal concerns. This document is written to outlast any specific runtime.

- **The complete ADR history** — every architectural decision made along the way.
  These live in docs/adr/. This document references their conclusions, not their reasoning.

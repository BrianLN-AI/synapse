# f(8): D-JIT Logic Fabric — Inference-Generated Candidates
**Version:** 1.0.0-Draft
**Status:** Design Specification
**Depends on:** f(7).md v1.0.0, f_7 (ContractCompliance gate, contract/definition blobs)

---

## 1. Problem Statement

After f_7, the fabric has a correctness gate: `ContractCompliance` (Pass 6) rejects any
blob whose behavior violates the promoted `contract/definition` for its label. The gate is
hard — a fast wrong blob cannot be promoted regardless of fitness.

What the fabric cannot do:
- **Author its own candidates** — every candidate in f_0–f_7 was hand-written by a human
- **Explore the implementation space** — evolution is bounded by what a human thought to write
- **Respond to fitness gaps** — if the planning blob underperforms on bursty workloads, a human
  must notice, understand why, and write a fix

f(8) introduces **inference-generated candidates**: the evolve engine calls an LLM with the
current blob payload, the behavioral contract, and live fitness signals. The LLM proposes an
implementation. The proposal enters the vault and faces the full governance chain — including
ContractCompliance — before it can be promoted.

The LLM has no special status. Its output is just another blob. The correctness gate, not the
human author, is what makes inference-generated candidates trustworthy.

---

## 2. The Authorship Shift

```
f_0 – f_7:  Human authors implementation → governance validates → vault admits
f_8:        LLM authors implementation  → governance validates → vault admits
```

The governance chain is **unchanged**. What changes is the source of candidates.

**Human role after f_8:** The human authors the *contract* — the pre/post specification that
defines what a correct implementation must do. The LLM implements against the spec. The
contract is the design; inference is the code generation step.

This is Design-by-Contract applied to machine authorship: the spec is expressed in code
(the contract/definition blob), and that code is the gate through which machine-generated
proposals must pass before they can run in production.

## 3. Inference Interface

### 3.1 `ai` Binary (Subprocess)

The evolve engine calls inference via the `ai` CLI binary — the same routing layer used
across all AI model access in this system. This satisfies C7 (AI model access routing):
use the `ai` binary, not direct API calls.

```python
# infer.py
import subprocess, os

result = subprocess.run(
    ["ai", "--no-daemon", "haiku", prompt],
    capture_output=True, text=True, timeout=60,
    env=os.environ,   # CF credentials from environment; never hardcoded
)
```

`--no-daemon` because the evolve engine should not own daemon lifecycle. Default model is
`haiku` (fast, low cost per candidate); the caller can pass a different model for quality tradeoff.

**Fallback:** if `ai` is not on `PATH` or returns a non-zero exit code, `evolve_one()` falls
back to the hand-authored mutation string for that label (the v8 implementations from f_7).
The system is never blocked on inference availability.

### 3.2 Credential Safety

Three Cloudflare AI Gateway credentials are required for inference routing:
- `CF_ACCOUNT_ID`
- `CF_GATEWAY_NAME`
- `CF_AIG_TOKEN`

These credentials:
1. **Live in the shell environment** — never in Python source, never in configuration files
2. **Are consumed by the `ai` binary** — evolve.py passes `env=os.environ` to subprocess;
   the credential values never appear in subprocess arguments (they are not `--token=...` flags)
3. **Never enter the vault** — the candidate blob payload contains only Python source; the audit
   log records only the candidate hash, not the generation prompt
4. **Are not logged** — audit events record `generated_via_inference: true`, not the prompt
5. **Are detectable if leaked** — the test suite scans vault and audit log for the literal strings
   `CF_ACCOUNT_ID`, `CF_GATEWAY_NAME`, `CF_AIG_TOKEN`, `Bearer`, and `cf-aig`

### 3.3 Generation Prompt

```
You are generating a candidate blob for the D-JIT Logic Fabric.

CURRENT IMPLEMENTATION:
```python
{current_payload}
```

BEHAVIORAL CONTRACT:
  pre:
{contract_pre_source}

  post:
{contract_post_source}

FITNESS SIGNALS:
  success_rate:     {success_rate:.3f}
  avg_latency_ms:   {avg_latency_ms:.2f}
  invocation_count: {invocation_count}

MUTATION GOAL: {mutation_goal}

ABI RULES (must be followed exactly):
  - `context` dict is the only input
  - `log()` callable is available for diagnostic output
  - `result` variable must be set before the blob exits
  - Only Python stdlib is available (no third-party imports)
  - No file system writes; no network calls; no subprocess

Return ONLY the Python source. No explanation, no markdown, no prose.
```

The contract pre/post source is included verbatim — the LLM sees the machine-checkable spec,
not paraphrased prose. This is the key difference from ungrounded generation: the LLM cannot
accidentally generate an implementation that "looks right" but violates the contract, because
the contract is specified in executable form and verified by Pass 6.

### 3.4 Response Parsing

The LLM response is stripped of markdown code fences if present:

```python
def _extract_source(response: str) -> str:
    match = re.search(r"```python\n(.*?)```", response, re.DOTALL)
    if match:
        return match.group(1).strip()
    match = re.search(r"```\n(.*?)```", response, re.DOTALL)
    if match:
        return match.group(1).strip()
    return response.strip()
```

The extracted source is PUT into the vault (`seed.put("logic/python", source)`) and then
reviewed by `triple_pass_review(hash, label=label)`. If Pass 6 rejects it, `evolve_one()`
raises `ReviewError` and the cycle records `"outcome": "review_failed"` — same as for any
other candidate that fails review.

## 4. Embedding (Vault Semantic Search)

Embedding is defined here for completeness. It is **optional in f_8** and may be deferred.

The `ai embed` command produces vector embeddings for blob payloads:

```bash
ai embed --batch texts.txt --model openai/text-embedding-3-small
```

Use cases within the evolve engine:
- **Seed selection**: find vault blobs semantically similar to the current baseline, to provide
  the LLM with additional implementation examples in the generation prompt
- **Test/case retrieval**: find promoted `test/case` blobs by semantic similarity to the label,
  supplementing the ContractCompliance probe context
- **Deduplication**: detect if a generated candidate is semantically identical to an existing
  vault blob (avoid generating the same implementation twice)

Embedding is decoupled from inference: the inference pipeline works without it. Embedding is
an optimization to the quality of the generation prompt, not a gate.

## 5. Evolution Cycle (Updated)

```
1. Read manifest: logic hash + contract hash
2. Load fitness signals from telemetry (telemetry-reader blob)
3. Generate candidate:
   a. Build prompt (current payload + contract + fitness signals + mutation goal)
   b. Call ai binary via subprocess
   c. Parse response → Python source
   d. If ai unavailable → fall back to hand-authored mutation string
4. seed.put("logic/python", candidate_source) → candidate_hash
5. triple_pass_review(candidate_hash, label=label)
   → Pass 1: StaticAnalysis
   → Pass 2: ResourceBounds
   → Pass 3: ProtocolCompliance (result must be set)
   → Pass 4: SemanticIntegrity
   → Pass 5: ReviewerIntegrity
   → Pass 6: ContractCompliance  ← hard correctness gate
6. benchmark(current_hash, candidate_hash, N rounds)
7. If candidate wins: issue_council_approval + promote(version="1.8.0")
8. Audit log: outcome, model used, generated_via_inference flag
```

---

## 6. Manifest Changes

No structural changes to the manifest schema. The version promoted by f_8 is `1.8.0`.

The audit log gains two new fields on `promote` events:
- `"generated_via_inference": true | false`
- `"inference_model": "haiku" | "sonnet" | null`

---

## 7. What f(8) Does Not Change

- The ABI contract is unchanged
- The vault structure is unchanged
- The pass numbering (Passes 1–6) is unchanged
- The reviewer chain is unchanged
- The manifest schema is unchanged
- Blobs promoted before f_8 are not affected
- ContractCompliance is not weakened — inference-generated candidates face the same gate as
  hand-authored ones; the gate is what makes inference safe

---

## 8. Forward: f(9)

f_8 establishes single-candidate inference: one prompt → one candidate → governance → promote
or reject. f(9) could introduce **multi-candidate generation**: generate N candidates per cycle,
rank by ContractCompliance pass rate and estimated fitness, select the most promising for
benchmarking. This is structurally similar to genetic algorithms applied at the blob level.

It could also introduce **guided mutation**: use fitness signal analysis to determine the
mutation goal automatically, rather than using a fixed string per label. The audit log already
captures enough signal to derive mutation goals programmatically.

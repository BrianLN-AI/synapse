# f(9): D-JIT Logic Fabric — Adversarial Test Authorship, Guided Evolution, Capability Blobs
**Version:** 1.0.0-Draft
**Status:** Design Specification
**Depends on:** f(8).md v1.0.0 (inference-generated candidates, ContractCompliance gate)

---

## 1. Problem Statement

After f_8, the fabric generates its own implementations. An LLM proposes a candidate;
ContractCompliance verifies it satisfies the behavioral contract; the governance chain admits
it if it wins the benchmark.

Three gaps remain:

**Gap 1 — Shared blind spots.** The same model that generates the implementation also
determines whether it "looks correct." A model that misunderstands the contract will generate
an implementation that satisfies its own misunderstanding. ContractCompliance catches behavioral
violations, but it probes with synthetic contexts. It does not generate adversarial inputs.
f(7).md §6 specifies that the tester reviewer must differ from the implementor reviewer — this
requirement has no enforcement mechanism yet.

**Gap 2 — Hardcoded mutation goals.** The mutation goal strings in evolve.py are static.
The evolve engine does not reason about what kind of mutation is most likely to improve fitness
given the current signal trends. If a blob's latency has been increasing over successive
promotions, the engine should infer "reduce latency" — not wait for a human to write that string.

**Gap 3 — Infrastructure only.** The fabric governs and evolves its own infrastructure blobs
(discovery, planning, telemetry-reader). These blobs make the fabric work — they are not
domain-specific capability. The fabric cannot yet be pointed at a problem and asked to solve it.

f(9) addresses all three, in order of dependency:

1. **Adversarial test authorship** — a tester model generates `test/case` blobs against the
   contract; the candidate must pass the test suite before benchmarking
2. **Guided mutation goals** — fitness signal trend analysis derives the mutation goal automatically
3. **Capability blobs + GOKR** — a new `goal/okr` blob type makes objectives first-class governed
   artifacts; capability blobs are domain-specific logic governed by the same chain as infrastructure

## 2. Adversarial Test Authorship (f_9)

### 2.1 Two-Model Pipeline

The evolve cycle gains a second inference pass:

```
1. Read manifest: logic hash + contract hash + test suite hashes
2. Implementor pass:  ai gemini-flash → candidate_payload
3. Vault: seed.put("logic/python", candidate_payload) → candidate_hash
4. Pass 1–6: triple_pass_review(candidate_hash, label=label)
5. Tester pass:   ai devstral → [test_case_payload_1, ..., test_case_payload_N]
6. Vault: seed.put("test/case", ...) for each → test_case_hashes
7. Approve + promote test cases: promote_test_cases(label, test_case_hashes, approval)
8. Test suite execution: run all promoted test/case blobs against candidate
   → if any test fails: ReviewError("TestSuiteFailed", ...)
9. Benchmark: old vs candidate (only reached if test suite passes)
10. Promote candidate if benchmark wins
```

The implementor and tester are different models by design. In f_9: `gemini-flash` (implementor)
and `devstral` (tester). The adversarial separation is structural — not a naming convention but
a physical fork in the inference pipeline.

### 2.2 Test Case Generation Prompt

The tester receives the contract and the candidate implementation, and is explicitly asked to
find failure modes:

```
You are an adversarial tester for the D-JIT Logic Fabric.

BEHAVIORAL CONTRACT for label '{label}':
  pre:
{contract_pre_source}

  post:
{contract_post_source}

CANDIDATE IMPLEMENTATION:
```python
{candidate_payload}
```

Your task: generate {N} adversarial test cases that probe boundary conditions,
edge cases, and inputs the implementor may not have considered.

For each test case, produce a JSON object:
{
  "for_label":     "{label}",
  "input_context": { ... },
  "expected":      { ... },
  "tolerance":     "exact | structural | behavioral",
  "rationale":     "one line: why this input is adversarial"
}

Return a JSON array of {N} test case objects. No prose, no markdown.
```

### 2.3 Test Suite Execution

Promoted `test/case` blobs are run against the candidate before benchmarking:

```python
def run_test_suite(label: str, candidate_hash: str, test_case_hashes: list[str]) -> list[dict]:
    results = []
    for tc_hash in test_case_hashes:
        tc_blob = seed._raw_get(tc_hash)
        tc = json.loads(tc_blob["payload"])
        if tc.get("for_label") != label:
            continue
        context = tc["input_context"]
        expected = tc.get("expected")
        tolerance = tc.get("tolerance", "behavioral")
        result = seed.invoke(candidate_hash, context)
        passed = _check_tolerance(result, expected, tolerance)
        results.append({"hash": tc_hash, "passed": passed, "result": result})
    return results
```

`tolerance` controls matching:
- `exact`: `result == expected`
- `structural`: same top-level keys (for dict results)
- `behavioral`: post-condition holds (defers to contract) — this is the default and most flexible

If any test fails, `evolve_one()` records `"outcome": "test_suite_failed"` and does not proceed
to benchmarking. The candidate is rejected. The test/case blobs that caused failure are logged
to the audit trail.

### 2.4 Test Case Promotion

`test/case` blobs are promoted via a new function `promote_test_cases()` parallel to
`promote_contract()`. They are stored in the manifest under:

```json
{
  "blobs": {
    "planning": {
      "logic/python":        "<hash>",
      "contract/definition": "<hash>",
      "test/case":           ["<hash>", "<hash>", ...]
    }
  }
}
```

The test suite accumulates across cycles: each f_9+ evolution run adds N new `test/case`
blobs to the label's test suite. Over time, the test suite becomes a comprehensive adversarial
corpus covering every implementation the evolve engine has ever generated.

### 2.5 Manifest Version

f_9 promotes to `1.9.0`.

## 3. Guided Mutation Goals (f_10)

### 3.1 Fitness Signal Trend Analysis

The audit log already contains benchmark events with fitness scores per cycle. f_10 adds a
`_derive_mutation_goal(label, audit_log)` function that reads recent benchmark events and
derives a goal:

```python
def _derive_mutation_goal(label: str, signals: dict) -> str:
    """Derive mutation goal from fitness signal trends in audit log."""
    entries = _read_audit_log()
    recent = [e for e in entries if e.get("label") == label and e.get("event") == "benchmark"][-5:]

    if not recent:
        return _MUTATION_GOALS[label]  # fallback to static goal

    # Trend: is latency increasing?
    latencies = [e["benchmark"]["new"]["fitness"] for e in recent if "benchmark" in e]
    # Derive from current signals
    sr  = signals.get("success_rate", 1.0)
    lat = signals.get("avg_latency_ms", 0.0)
    vol = signals.get("volatility", 0.0)

    if sr < 0.8:
        return f"Improve success rate for {label}: currently {sr:.1%}, failures must be reduced."
    if vol > 0.5:
        return f"Reduce latency volatility for {label}: volatility={vol:.2f}, prioritize consistency over speed."
    if lat > 10.0:
        return f"Reduce avg latency for {label}: currently {lat:.1f}ms, optimize the hot path."
    return f"Improve overall fitness for {label}: explore alternative algorithms."
```

This replaces `_MUTATION_GOALS` with a signal-driven derivation. The hardcoded strings become
fallbacks for when the audit log has insufficient data.

### 3.2 Multi-Candidate Generation

f_10 also generates N candidates per cycle and selects the one most likely to win:

```
for i in range(N_CANDIDATES):
    candidate_payload = infer.generate_candidate(..., mutation_goal=derived_goal)
    candidate_hash    = seed.put("logic/python", candidate_payload)
    try:
        triple_pass_review(candidate_hash, label=label)  # Pass 1–6
        run_test_suite(label, candidate_hash, test_suite)
        candidates.append(candidate_hash)
    except (ReviewError, TestSuiteFailed):
        continue  # discard failing candidates

# Benchmark all passing candidates against current
# Promote the one that wins
```

`N_CANDIDATES = 3` initially. The cost is 3 inference calls per cycle — acceptable for
a governed promotion gate.

## 4. Capability Blobs and GOKR Execution (f_11)

### 4.1 The Insight: The Evolve Cycle IS a GOKR Loop

The fabric's evolve cycle already has the structure of a GOKR loop:

| GOKR concept | Fabric equivalent |
|---|---|
| Objective | Mutation goal (what we want to improve) |
| Key Results | Fitness signals (success_rate, latency, volatility) |
| Initiatives | Candidate blobs (implementations toward the KR) |
| Iteration | Benchmark cycle (measure, promote winners, discard losers) |
| Retrospective | Audit log (every cycle recorded, trends derivable) |

What is missing: the **Objective** is not a governed artifact. It lives in a Python string in
evolve.py. Key Results are not declared; they are inferred from fitness signal thresholds. The
GOKR structure is real but ungoverned.

### 4.2 `goal/okr` Blob Type

A new blob type that governs the objective for a capability:

```json
{
  "for_label":   "planning",
  "objective":   "Select the highest-quality blob with the lowest operational cost",
  "key_results": [
    {
      "metric":    "success_rate",
      "target":    0.95,
      "direction": "above"
    },
    {
      "metric":    "avg_latency_ms",
      "target":    2.0,
      "direction": "below"
    },
    {
      "metric":    "volatility",
      "target":    0.3,
      "direction": "below"
    }
  ],
  "supersedes": null
}
```

The `goal/okr` blob is governed: Triple-Pass + Council Approval before it can drive evolution.
The objective field is human-authored prose. The key results are machine-checkable — each maps
to a fitness signal with a target and direction.

When a `goal/okr` is promoted for a label, `_derive_mutation_goal()` reads the key results and
current signal values to generate the mutation goal:

```python
def _mutation_goal_from_okr(label: str, okr: dict, signals: dict) -> str:
    gaps = []
    for kr in okr.get("key_results", []):
        metric   = kr["metric"]
        target   = kr["target"]
        direction = kr["direction"]
        current  = signals.get(metric)
        if current is None:
            continue
        if direction == "above" and current < target:
            gaps.append(f"{metric}={current:.3f} (target: >{target})")
        elif direction == "below" and current > target:
            gaps.append(f"{metric}={current:.3f} (target: <{target})")
    if not gaps:
        return f"All key results met for {label} — explore robustness improvements."
    return (f"Improve {label} to close key result gaps: " + ", ".join(gaps) + ". "
            f"Objective: {okr['objective']}")
```

The mutation prompt sent to the LLM now includes the objective verbatim. The LLM generates
toward the declared intent, not toward a hardcoded string.

### 4.3 Capability Blobs

A capability blob is a `logic/python` blob governed under a user-defined label — not one of
the three infrastructure labels (discovery, planning, telemetry-reader), but a domain-specific
one:

```python
# Example: a "summarizer" capability
seed.put("logic/python", """
document = context.get("document", "")
max_words = context.get("max_words", 100)
# ... implementation ...
result = {"summary": ..., "word_count": ...}
""")
```

To govern a capability, you:
1. Write a `contract/definition` blob for the label
2. Write a `goal/okr` blob for the label
3. Call `bootstrap_capability(label, contract_hash, okr_hash, initial_blob_hash)`

The fabric then evolves the capability blob autonomously:
- Inference generates candidates against the contract + OKR goal
- ContractCompliance verifies correctness
- The adversarial tester generates test cases
- The benchmark measures fitness
- Promotions drive the implementation toward the key results

The human role: write the contract (what it must do) and the OKR (what success looks like).
The fabric handles the rest.

### 4.4 Manifest Extension

```json
{
  "version": "1.11.0",
  "blobs": {
    "planning": {
      "logic/python":        "<hash>",
      "contract/definition": "<hash>",
      "test/case":           ["<hash>", ...],

## 5. The Complete Picture: What the Fabric Becomes

After f_9, f_10, f_11:

```
Human provides:
  - contract/definition: what the blob must do (machine-checkable pre/post)
  - goal/okr:            what success looks like (objective + measurable key results)

Fabric does autonomously:
  - Generates candidate implementations (implementor model)
  - Generates adversarial test suites (tester model, different from implementor)
  - Verifies correctness (ContractCompliance, test suite execution)
  - Derives mutation goals from OKR gaps
  - Benchmarks and promotes winners
  - Accumulates test corpus across every cycle
  - Records all decisions in the audit log, traceable to the trust root
```

This is the GOKR execution engine: you write the spec and the success criteria; the fabric
authors, validates, and evolves implementations toward them.

The analogy to product teams: the contract is the PRD (what it does), the OKR is the success
metric, the fabric is the engineering team. The difference: no meetings, no handoffs, full
auditability.

## 6. Implementation Sequence

| Version | What it adds |
|---------|-------------|
| f_9  (v1.9.0)  | Adversarial test authorship (devstral tester), test suite execution gate, test/case promotion |
| f_10 (v1.10.0) | Guided mutation goal derivation from audit log trends, multi-candidate generation (N=3) |
| f_11 (v1.11.0) | `goal/okr` blob type, capability bootstrap, full GOKR execution loop |

Each version is buildable independently. f_9 does not require f_10 or f_11.

---

## 7. What Does Not Change

- The ABI contract (context → result) is unchanged
- The vault structure is unchanged
- Pass 1–6 numbering is unchanged
- The trust root bootstrapping mechanism is unchanged
- Infrastructure blobs (discovery, planning, telemetry-reader) remain — they govern
  the fabric's own operation and are evolved the same way as capability blobs
- The `ai` binary routing pattern is unchanged

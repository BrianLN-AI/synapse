# (λ) — The D-JIT Logic Fabric Foundation

**Version:** 1.0 — 2026-04-02
**Status:** Stable invariants

This document describes the D-JIT Logic Fabric at its most abstract. It is the spec
that travels — sufficient to implement the system in any language, on any runtime,
by anyone who has only this document and a blake3 implementation (see test vectors in KEY).

---

## The six invariants

```
KEY:        multihash(envelope) = address   [self-describing: function_id | length | digest]
LEDGER:     manifest — self-sealing, self-describing, tenant-scoped
BOOTSTRAP:  engine_hash = manifest.engine; invoke = exec(vault.get(engine_hash))
INTERFACE:  expression(context) → {result, attestation}
TELEMETRY:  every invocation writes an expression; feedback anchors to telemetry hash
GATE:       generate → test → fitness → governance → promote; nothing is current without this
```

A correct implementation satisfies all six. Nothing else is required.

## KEY: multihash address format

An expression's address IS its hash. Identity is derived, not assigned. No registry.
The address is self-describing — it carries its own decoding instructions.

```
envelope = { "type": "<namespace/kind>", "payload": "<content>" }
address  = function_id | digest_length | digest_bytes
```

**String form** (canonical, human-readable):
```
<function-prefix>:<hex-digest>

blake3:af1349b9f5f9a1a6a0404dea36dcc9499bcb25c9adc112b7cc9a93cae41f3262
sha256:e3b0c44298fc1c149afbf4c8996fb924...
poseidon:7f3c21...
```

**Canonical function codes** (IPFS multicodec table):

| Function | Prefix | Code | Use |
|---|---|---|---|
| blake3 | `blake3` | 0x1e | Vault addressing (canonical) |
| sha2-256 | `sha256` | 0x12 | Legacy / interop |
| poseidon-bls12_381 | `poseidon` | 0xb401 | ZK circuits (low constraint count) |

The type is part of the preimage. Changing the type produces a different expression at
a different address. Two expressions with identical content at identical types are the
same expression — stored once, referenced by the same hash everywhere.

**Vault hash vs ZK hash — two separate concerns:**

```
vault_address = blake3(envelope)      fast, widely available, general-purpose
zk_address    = poseidon(content)     ZK-native, ~10x fewer in-circuit constraints
```

When they diverge, a hash-bridge expression in the vault links them:
```json
{
  "type": "meta/hash-bridge",
  "payload": {
    "vault_address": "blake3:af1349b9...",
    "zk_address":    "poseidon:7f3c21...",
    "proof":         "<zk-proof-that-both-hash-same-content>"
  }
}
```

**The bootstrap property:** a new party reads any address, extracts the function_id,
looks up the algorithm, verifies the content. No external context required.

**Canonical blake3 test vectors** [MEASURED — Python blake3 package, 2026-04-02]:
```
blake3(b"")              = af1349b9f5f9a1a6a0404dea36dcc9499bcb25c9adc112b7cc9a93cae41f3262
blake3(b"abc")           = 6437b3ac38465133ffb63b75273a8db548c558465d79db03fd359c6cd5bd9d85
blake3(bytes(range(251)))= 2a43e6bf5d7dfe202bf9653c94aacb221a20cd5e449602684d9ffbd38d9a8920
```

Verify your blake3 implementation against these before building anything else.

**Blake3 specification:** O'Connor et al., "BLAKE3 — One Function, Fast Everywhere" (2020).
Reference implementation: https://github.com/BLAKE3-team/BLAKE3

**What is invariant:** `address = multihash(content)` — the self-describing format.
**What is swappable:** the specific function_id, provided all parties in a federation agree.
Backward compat: bare 64-char hex addresses are treated as implicit `blake3:<hex>`.

## LEDGER: manifest

The manifest is the current-state registry. It maps labels to expression hashes:

```json
{
  "tenant":     "<genesis-expression-hash>",
  "governance": "<governance-expression-hash>",
  "blobs": {
    "<label>": { "<type>": "<expression-hash>" }
  },
  "version": "<semver>",
  "hash":    "<blake3-of-manifest-without-hash-field>"
}
```

**Self-sealing:** `manifest.hash = blake3(manifest excluding the hash field)`. The manifest
verifies its own integrity. No external validator needed.

**Self-describing:** the manifest contains the engine expression hash. Given the manifest
and the vault, any party can reconstruct the system from scratch.

**Tenant-scoped:** each tenant has one manifest. Multiple tenants share the vault.

**What is invariant:** the self-sealing property (hash field excluded from preimage).
**What is swappable:** format (JSON, CBOR, etc.), provided serialization is deterministic.

## BOOTSTRAP: engine as expression

```
manifest    = load("manifest.json")
engine_hash = manifest.governance_and_blobs["engine"]["logic/<runtime>"]
payload     = vault.get(engine_hash).payload
invoke      = primitive_exec(payload)    ← ONE fixed exec/eval/dlopen call
```

`primitive_exec` is the single fixed dispatch in the system. It is called exactly once:
to load the engine expression. All subsequent invocations go through the derived `invoke`.

The engine expression contains execution policy: telemetry wrapping, ABI enforcement,
bytecode caching, sandboxing. These are not in the kernel. The kernel is: vault + manifest
loader + one primitive dispatch. Everything above that is the engine expression's payload,
subject to governance like any other expression.

**What is invariant:** the engine is a content-addressed expression; there is ONE fixed
dispatch to load it; all subsequent execution is derived.
**What is swappable:** `primitive_exec` implementation (`exec` in Python, `eval` in JS,
`instantiate` in WASM, etc.).

## ABI: the calling convention

```
expression(context) → { result, attestation }
```

- `context` — input map (may be empty `{}`)
- `result` — output value (any non-null); absence is a hard error (ABI violation)
- `log(msg)` — side channel injected by engine, captured to telemetry
- `attestation` — governance status: `{tier: 1|2|3, governance: hash|null, proof: bytes|null}`

**Attestation tiers:**
- Tier 1: executed by hash, no governance claim
- Tier 2: governance-recognized via alliance
- Tier 3: your governance promoted this expression

The engine does not refuse to execute Tier 1 expressions. Attestation is a witness, not
a gate. Callers decide whether to require a minimum tier.

**What is invariant:** the shape — one input map, one output value, result-absence is an
error, log side channel. This is the interoperability contract. Blobs from one
implementation run on another if and only if they speak the same ABI.
**What is swappable:** variable names (`context`/`result` are conventions), attestation
format details.

## TELEMETRY: invocations as expressions

Every invocation writes a telemetry expression to the vault:

```json
{
  "type": "telemetry/json",
  "payload": {
    "logic_hash":   "<expression-hash>",
    "latency_ms":   42.3,
    "memory_kb":    128,
    "log_lines":    ["..."],
    "error":        null,
    "timestamp":    "<ISO-8601>",
    "tenant":       "<tenant-id>"
  }
}
```

Feedback anchors to a telemetry hash, not a logic hash directly. This proves which
execution the feedback came from — no fabrication possible without the actual execution
record.

**What is invariant:** feedback must be anchored to a proven execution record.
**What is swappable:** telemetry storage (vault, separate log store), format.

## GATE: nothing is current without passing this

```
generate(current, contract, fitness_target)   → candidate_hash
test(candidate_hash, test_expressions)        → pass | fail
score(telemetry)                              → fitness
governance_approve(candidate_hash)            → approval_hash
promote(label, candidate_hash, approval_hash) → manifest_version
```

No expression becomes "current" in the manifest without completing this sequence.
The approval artifact is a content-addressed expression in the vault — permanent,
auditable, referenced by the promoted manifest version.

**Governance types** (ADR-013):
- `governance/single-key` — one key, one signature (bootstrap)
- `governance/quorum` — N of M threshold
- `governance/threshold` — automated fitness floor
- `governance/proof` — ZK proof of fitness

**What is invariant:** there IS a gate; nothing is current without passing it.
**What is swappable:** specific governance type, fitness formula, test criteria.

## The vault as Merkle DAG

The vault is not a flat key-value store. It is a Merkle DAG — a directed acyclic graph
where every node is content-addressed. Beyond direct hash references embedded in payloads
(acyclic by construction), the graph has typed edges stored as expression artifacts:

```
GOVERNS  — governance artifact → promoted expression
OBSERVES — telemetry record → logic expression
PRECEDES — expression → prior version
DEPENDS  — expression → referenced expression
TAGGED   — tag artifact → expression
SIMILAR  — semantic proximity (embedding-derived)
```

Chains (totally ordered sequences) are projections over one edge dimension. A governance
approval chain, a lineage chain, a telemetry observation chain — all emerge from the CHAIN
traversal strategy applied to the relevant edge type. An external blockchain is a chain
projected over a consensus subgraph; the vault's own Merkle DAG provides equivalent
tamper-evidence without requiring external consensus.

## Tenant identity

```json
{
  "type": "governance/genesis",
  "payload": {
    "public_key": "<ed25519-hex>",
    "governance": "<initial-governance-expression-hash>",
    "name":       "<human-readable>",
    "founded":    "<ISO-8601>"
  }
}
```

`tenant_id = blake3(genesis_envelope)` — identity derived from content, no central registry.
The genesis blob is content-addressed. To verify ownership: retrieve genesis blob at
`tenant_id`, find the public key, verify signature.

Multiple tenants share the vault. Each tenant has their own manifest. Two tenants who
independently produce the same expression store one copy — deduplication is free and
automatic.

## The bootstrap sequence (complete)

```
1. Load manifest.json
2. Verify: manifest.hash == blake3(manifest excluding hash field)
3. Find engine: manifest.blobs.engine["logic/<runtime>"]
4. Load engine payload: vault.get(engine_hash).payload
5. ONE primitive dispatch: invoke = exec(engine_payload)
6. All subsequent execution: invoke(hash, context)
```

Given only this document, blake3, and a filesystem, any implementation can execute
step 1–6 and reach a running system. The engine expression determines all subsequent
behavior.

---

## What makes an implementation correct

An implementation is correct if:
1. `vault.get(vault.put(type, payload)) == {type, payload}` — round-trip fidelity
2. `vault.get(h).hash == h` for all stored expressions — content addressing holds
3. `manifest.hash == blake3(manifest excluding hash field)` — self-sealing holds
4. `invoke(h, {}) raises ABI_VIOLATION` when expression does not set `result`
5. Every invocation writes a telemetry expression containing `logic_hash`, `latency_ms`
6. No expression becomes current in the manifest without a governance approval artifact
   in the vault referencing it

Two correct implementations can exchange expression hashes and invoke each other's
promoted expressions, provided they agree on the hash function and ABI.

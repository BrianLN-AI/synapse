# f(7): D-JIT Logic Fabric Correctness Layer
**Version:** 1.0.0-Draft
**Status:** Design Specification
**Depends on:** f(undefined).md v1.3.0, f_6 (council/reviewer governance chain)

---

## 1. Problem Statement

After f_6, the fabric has **governed selection**: it selects among pre-authored blob candidates,
governs their admission through Triple-Pass + Council Approval, and measures their fitness
through telemetry + feedback. The reviewer chain is complete — every promotion is traceable
back to a content-addressed trust root.

What the fabric cannot do:
- **Generate** its own candidates (it selects, it does not author)
- **Verify correctness** of an implementation beyond "it set `result` and didn't crash"
- **Reason about intent** — there is no governed artifact for what a blob is *supposed* to do

f(7) introduces the **Correctness Layer**: governed artifacts that define, contract, and
critique blob behavior. This is the prerequisite for safe inference-generated candidates.
Without a correctness gate, a fast blob that silently computes the wrong answer looks
identical to a correct one in the fitness formula.

---

## 2. Core Principle: Design-by-Contract

Every promotable blob may carry a `contract/definition` blob that declares its behavioral
contract in executable form:

```python
def pre(context: dict) -> bool:
    """Pre-condition: what must be true about context before invocation."""
    ...

def post(context: dict, result) -> bool:
    """Post-condition: what must be true about result after invocation."""
    ...

def invariant(context: dict, result) -> bool:
    """Invariant: properties that must hold regardless of inputs."""
    ...
```

The contract IS the design specification. It is precise, executable, and machine-checkable.
It is not prose — it is code that fails deterministically when violated.

The existing ABI contract (`result` must be set) is a post-condition already enforced by
Pass 3 (ProtocolCompliance). `contract/definition` extends this mechanism rather than
replacing it.

## 3. New Blob Types

### 3.1 `contract/definition`
Declares the behavioral contract for a specific `logic/python` blob.

Required fields in payload (JSON):
```json
{
  "for_label": "planning",
  "pre":        "<python source: def pre(context) -> bool>",
  "post":       "<python source: def post(context, result) -> bool>",
  "invariant":  "<python source: def invariant(context, result) -> bool | null>"
}
```

- Authored by the **designer** reviewer role
- Governed: Triple-Pass + Council Approval before use
- Once promoted, the evolve engine requires a candidate to satisfy the contract before promotion
- Different versions of a capability may have different contracts (contracts evolve with the spec)

### 3.2 `test/case`
Structured input/output expectation for a logic blob. Authored by the **tester** reviewer
role — adversarially, to find inputs the implementor did not consider.

Required fields in payload (JSON):
```json
{
  "for_label":      "planning",
  "input_context":  { ... },
  "expected":       { "selected": "...", ... },
  "tolerance":      "exact | structural | behavioral",
  "rationale":      "why this case exercises a non-obvious path"
}
```

- `tolerance` controls matching: `exact` (result == expected), `structural` (same keys),
  `behavioral` (post-condition holds — defers to contract)
- A promoted test suite is required before inference-generated candidates can be promoted
- Adversarial authorship: the tester reviewer and implementor reviewer must differ

### 3.3 `adr/decision`
Architecture Decision Record — the governed rationale for a design choice.

Required fields in payload (JSON):
```json
{
  "for_label":    "telemetry-reader",
  "decision":     "Use Welford online variance instead of stored latency lists",
  "context":      "...",
  "alternatives": ["store all values (O(n) memory)", "fixed window (loses history)"],
  "consequences": "O(1) memory per blob; p95 is an approximation, not exact",
  "supersedes":   "<prior adr/decision hash | null>"
}
```

- Immutable — you do not edit an ADR, you publish a new one that `supersedes` the prior
- The `supersedes` chain is auditable through the vault
- The manifest links the current ADR hash per label

### 3.4 `critique/finding`
Output of the critic reviewer role: gaps identified in a contract, test suite, or
implementation. Feeds back into the design cycle.

Required fields in payload (JSON):
```json
{
  "for_label":   "discovery",
  "target_hash": "<contract/definition or test/case hash being critiqued>",
  "finding":     "...",
  "severity":    "blocking | advisory",
  "suggestion":  "..."
}
```

- `blocking` findings prevent promotion until addressed
- `advisory` findings are recorded but do not block

## 4. Extended Triple-Pass Review

### Pass 6: ContractCompliance

Applies to `logic/python` blobs when a promoted `contract/definition` exists for their label.

Protocol:
1. Load the promoted `contract/definition` blob for the label
2. Compile the `pre`, `post`, and `invariant` functions from the contract payload
3. Invoke the logic blob with the contract's sample inputs (drawn from promoted `test/case`
   blobs for the label, if present; otherwise a synthetic probe context)
4. For each invocation: verify `pre(context)` held before, `post(context, result)` holds
   after, `invariant(context, result)` holds throughout
5. Fail `ContractCompliance` if any condition is violated

ContractCompliance is a **hard gate**: a blob that violates its contract cannot be promoted,
regardless of fitness. Fitness measures performance. ContractCompliance measures correctness.
These are orthogonal concerns.

**Bootstrap note:** Blobs promoted before a contract exists for their label are not
retroactively subject to ContractCompliance. The pass only fires when a promoted contract
is present in the manifest for that label.

## 5. Extended Manifest Structure

Each label entry in `manifest["blobs"]` is extended:

```json
{
  "blobs": {
    "planning": {
      "logic/python":        "<hash>",
      "contract/definition": "<hash | null>",
      "test/case":           ["<hash>", ...],
      "adr/decision":        "<hash | null>"
    }
  }
}
```

Promotion of `contract/definition`, `test/case`, and `adr/decision` blobs follows the
same path as logic blobs: Triple-Pass + Council Approval signed by the appropriate reviewer.

## 6. Four-Role Reviewer Model

The council/reviewer registry (f_6) is extended with four canonical roles. Each role is
a governed `council/reviewer` blob with distinct `authorized_types` and `trust_weight`.

| Role | authorized_types | trust_weight | Concern |
|------|-----------------|--------------|---------|
| designer | `contract/definition`, `adr/decision` | 1.0 | What the blob must do and why |
| implementor | `logic/python` | 0.8 | How the blob does it |
| tester | `test/case` | 1.0 | What inputs reveal failure modes |
| critic | `critique/finding` | 0.9 | Gaps between contract, tests, and implementation |

**Adversarial separation:** The tester reviewer must differ from the implementor reviewer.
A `test/case` blob approved by the same reviewer that approved the implementation is worth
less — shared authorship means shared blind spots. The `reviewer_trust` weighting in the
telemetry reader already encodes this: tester authority on test cases, implementor authority
on logic, designer authority on contracts.

The critic reviewer operates across all types: given a contract, a test suite, and an
implementation, it produces `critique/finding` blobs that identify gaps. A critic finding
a gap in the test suite against the contract is a higher-signal finding than a critic
finding an implementation note.

## 7. Inference Integration Pattern (Forward: f_8)

f(7) establishes the correctness layer. f(8) introduces inference-generated candidates.
The integration pattern is defined here so f(7) is built to support it.

**Generation is outside the vault. Admission is fully governed.**

```
evolve engine (Python)
  → calls LLM with: current_payload + contract + fitness_signals + mutation_goal
  → receives: candidate_payload (Python source)
  → seed.put("logic/python", candidate_payload)          ← enters vault
  → triple_pass_review(candidate_hash)                   ← Pass 1–6, including ContractCompliance
  → benchmark(current_hash, candidate_hash)
  → promote(...)                                         ← Council Approval required
```

The LLM has no special status. Its output is just another blob. The correctness gate
(ContractCompliance) is what makes inference-generated candidates trustworthy enough to
benchmark. Without f(7), inference-generated candidates could pass all review passes and
still compute the wrong answer.

**Embedding and reranking (also f_8 or later):**
Embedding gives the vault semantic search: "find blobs similar to this one." Reranking
refines approximate embedding search with a more expensive cross-encoder model. Both are
useful once the candidate pool is large enough for approximate ranking to introduce noise.
Neither is required for f(7).

## 8. Evolution Cycle with Correctness Layer

The f_7 evolution cycle adds two steps before benchmarking:

```
1. Read current manifest: logic hash + contract hash + test suite hashes
2. Generate/author candidate logic blob (hand-authored in f_7; inference-generated in f_8)
3. Triple-Pass Review (Pass 1–5 existing + Pass 6 ContractCompliance if contract present)
4. Run promoted test suite against candidate (all test cases must pass)
5. Benchmark: old vs new, N rounds, derive tolerance from audit log
6. If candidate wins: Council Approval (designer or implementor reviewer) → promote
7. Write benchmark + contract compliance result to audit log
```

The manifest promotion in f_7 targets version `1.7.0`.

## 9. What a Capability Is (Complete Picture)

A fully-specified capability in the vault after f(7):

| Artifact | Blob type | Answers |
|----------|-----------|---------|
| Implementation | `logic/python` | How it works |
| Contract | `contract/definition` | What it must do (machine-checkable) |
| Test suite | `test/case` (N blobs) | What inputs stress it |
| Decision record | `adr/decision` | Why it was designed this way |
| Critique findings | `critique/finding` (N blobs) | What gaps were identified |
| Telemetry | `telemetry/artifact` (N blobs) | How it has performed |
| Feedback | `feedback/outcome` (N blobs) | What downstream systems observed |

All content-addressed. All immutable. All linked through the manifest. All governed by
the same reviewer chain back to the trust root.

## 10. What f(7) Does Not Change

- The ABI contract (context in, log callable, result out) is unchanged
- The vault structure (flat directory, SHA-256 content addressing) is unchanged
- The existing pass numbering (Passes 1–5) is unchanged; ContractCompliance is Pass 6
- The trust root bootstrapping mechanism is unchanged
- Blobs without a promoted contract are unaffected — ContractCompliance is a no-op if
  no contract is present for the label
